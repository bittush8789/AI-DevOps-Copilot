🤖 AWS Bedrock Integration

Model Configuration

# bedrock_service.py
client = boto3.client(
    service_name="bedrock-runtime",
    region_name=settings.AWS_REGION,
)

body = {
    "anthropic_version": "bedrock-2023-05-31",
    "max_tokens": 4096,
    "temperature": 0.7,
    "messages": [{"role": "user", "content": prompt}],
    "system": system_prompt,
}

response = client.invoke_model(
    modelId="anthropic.claude-3-sonnet-20240229-v1:0",
    body=json.dumps(body),
)

Authentication Priority

  1. Bedrock API Key – set AWS_BEDROCK_API_KEY; mapped internally to AWS_BEARER_TOKEN_BEDROCK (boto3 env var)
  2. IAM Credentials – set AWS_ACCESS_KEY_ID + AWS_SECRET_ACCESS_KEY as fallback
  3. IRSA (Production) – EKS pod identity; no credentials stored in cluster

✅ Zero-credential local dev

Generate a Bedrock API key (long-term up to 30d or short-term up to 12h) from the AWS Console → Bedrock → API Keys, then export AWS_BEDROCK_API_KEY=<key>

🚀 One-Command Local Deploy

export AWS_BEDROCK_API_KEY=<your-bedrock-api-key>
./scripts/deploy-kind.sh

What the script does

  1. Creates a Kind cluster named ai-devops with port mappings 3000→30080 and 8000→30081
  2. Builds Docker images for backend (FastAPI) and frontend (React/Nginx)
  3. Loads images directly into the Kind node (no registry needed)
  4. Runs kubectl apply -k k8s/overlays/kind — creates namespace, ConfigMap, RBAC, services, and deployments
  5. Creates K8s Secrets: bedrock-credentials (from AWS_BEDROCK_API_KEY)
  6. Waits for deployments to be Available (kubectl wait --timeout=300s)
  7. Patches services to NodePort for localhost access

Security Architecture

🔐 Security Model

Authentication Tiers

🔑

Bedrock API Key (Dev)

Short-term (≤12h) or long-term (≤30d) key from AWS Console. Set via AWS_BEDROCK_API_KEY; no IAM setup needed

👤

IAM Credentials (CI/CD)

AWS_ACCESS_KEY_ID + AWS_SECRET_ACCESS_KEY. Stored as K8s secrets; optional fallback

🏭

IRSA (Production EKS)

IAM Role for Service Account. Pod identity; zero credentials stored in cluster — recommended for production

Additional Security Controls

RBAC

ServiceAccount with least-privilege ClusterRole; only read access to pods and logs

Network Policy

Ingress/Egress restricted; backend only reachable from frontend namespace

Non-root Container

Dockerfile creates appuser (UID 1000); runs with read-only root filesystem

Secrets Management

All credentials injected as K8s Secrets (never in ConfigMaps or env literals)

📝 Implementation Lessons Learned

✅

Bedrock API Keys

Use AWS_BEDROCK_API_KEY for quick setup; maps to AWS_BEARER_TOKEN_BEDROCK recognized by boto3. No IAM config needed for local dev

⚠️

K8s Env Var Conflicts

Kubernetes auto-injects service URLs as <SERVICE>_PORT=tcp://.... Avoid naming config vars after service names (e.g., use APP_PORT not BACKEND_PORT)

📦

Dependency Pinning

Pin boto3≥1.34.72 for langchain-aws compatibility. Use requests-aws4auth for OpenSearch AWS auth

🔄

Kustomize Generators

Use behavior: replace (not merge) in overlay configMapGenerator when base has a static ConfigMap, or use patch files instead

🐳

Docker Build

Use npm install (not npm ci) when package-lock.json is absent. Add vite-env.d.ts for import.meta.env TypeScript support

🔑

IRSA vs API Keys

IRSA (production) = zero-credential pods. API keys = easiest for local dev. Never store long-lived IAM keys in container images
